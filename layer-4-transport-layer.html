Layer 4: The Transport Layer

Introduction

Layers 1 through 3 are responsible for getting data physically from one device to another — but they do not care which application that data belongs to, or whether it arrived correctly and in order. That is the job of Layer 4 — the Transport Layer.

The Transport Layer manages end-to-end communication between applications running on two different hosts. It decides how data is broken up for sending, how it gets reassembled on the other end, and whether delivery needs to be guaranteed or not.

Real-world example Imagine loading a webpage while streaming music at the same time. Both use the same internet connection, but the Transport Layer uses port numbers to keep those two data streams completely separate so each application only receives the data meant for it.

Transport Layer Services

The Transport Layer provides several key services that sit between the network and your applications:

  • End-to-end delivery: Ensures data gets from the sending application all the way to the receiving application — not just to the destination device
  • Addressing via port numbers: Uses port numbers to direct data to the correct application (e.g., port 80 for HTTP, port 443 for HTTPS)
  • Reliable delivery (TCP): Optionally guarantees that all data arrives, in order, with no missing pieces
  • Flow control: Prevents a fast sender from overwhelming a slow receiver by regulating how much data can be in transit at once
  • Multiplexing: Allows multiple applications to share the same network connection simultaneously using different port numbers

Port Numbers

A port number is a 16-bit number (0–65535) that identifies a specific application or service on a device. While an IP address gets data to the right machine, the port number gets it to the right application on that machine.

Think of it like an apartment building: the IP address is the building's street address, and the port number is the specific apartment number inside.

Well-Known Ports

Ports 0–1023 are reserved for common, well-known services:

  • Port 20/21 — FTP (file transfer)
  • Port 22 — SSH (secure remote login)
  • Port 23 — Telnet (unencrypted remote login, legacy)
  • Port 25 — SMTP (sending email)
  • Port 53 — DNS (domain name resolution)
  • Port 67/68 — DHCP (automatic IP address assignment)
  • Port 80 — HTTP (unencrypted web traffic)
  • Port 443 — HTTPS (encrypted web traffic)
Exam tip Memorizing well-known port numbers is a guaranteed topic on both Network+ and CCNA. At minimum, know ports 20/21 (FTP), 22 (SSH), 23 (Telnet), 25 (SMTP), 53 (DNS), 80 (HTTP), and 443 (HTTPS).

TCP vs. UDP

The Transport Layer has two primary protocols: TCP (Transmission Control Protocol) and UDP (User Datagram Protocol). They serve very different purposes, and choosing the right one is a fundamental networking decision.

TCP

  • Connection-oriented
  • Guaranteed transmission
  • Three-way handshake
  • Examples: HTTP, HTTPS, FTP

UDP

  • Connectionless
  • No guarantee
  • Lower latency
  • Examples: DNS, DHCP, TFTP

When to Use TCP

TCP is used when accuracy matters more than speed. It guarantees that every byte arrives at the destination in the correct order. If a packet is lost, TCP detects this and requests it be re-sent. This makes TCP ideal for:

  • Loading webpages (HTTP/HTTPS) — every byte of HTML, CSS, and JavaScript must arrive correctly
  • Sending emails (SMTP) — a corrupted email is useless
  • Downloading files (FTP) — a missing chunk corrupts the entire file

When to Use UDP

UDP is used when speed matters more than perfection. It sends data without establishing a connection and without waiting for acknowledgment. A few lost packets are acceptable when:

  • Streaming video or audio — a missing frame causes a brief glitch, not a broken experience
  • Online gaming — old position data is useless; you need the newest update as fast as possible
  • DNS lookups — small requests that fit in a single packet and can simply be re-sent if lost
  • VoIP calls — a slight audio blip is better than noticeable delay caused by TCP retransmission
Key distinction TCP trades speed for reliability. UDP trades reliability for speed. Neither is better — they are tools designed for different jobs. Understanding which protocol an application uses, and why, is a core networking concept.

The TCP Three-Way Handshake

Before TCP transfers any data, it must first establish a connection through a process called the three-way handshake — a brief exchange of three messages confirming both sides are ready to communicate.

  1. SYN — The client sends a synchronization request to the server: "I want to connect. Here is my starting sequence number."
  2. SYN-ACK — The server acknowledges the request and sends back its own sequence number: "Got it. I am ready. Here is mine."
  3. ACK — The client acknowledges the server's response: "Confirmed. Let's begin."

Only after this handshake completes does actual data transfer begin. This setup ensures both parties know each other's sequence numbers, which TCP uses to track and reassemble data in the correct order.

Why sequence numbers matter TCP breaks large data into many small segments, each numbered in sequence. If segments arrive out of order — which is common across the internet — the receiver uses those numbers to reassemble them correctly before passing the data up to the application.

Flow Control & Windowing

Flow control prevents a fast sender from overwhelming a slow receiver. TCP handles this through a mechanism called windowing: the receiver tells the sender how much buffer space it currently has available (the window size), and the sender limits how much data it sends before waiting for an acknowledgment.

If the receiver is getting overwhelmed, it advertises a smaller window size — effectively telling the sender to slow down. As buffer space frees up, it advertises a larger window and throughput increases again. This process happens automatically and continuously throughout every TCP session.

Key Takeaways

Layer 4 — Transport

  • Manages end-to-end delivery between applications
  • Uses port numbers to identify services
  • Two protocols: TCP (reliable) and UDP (fast)
  • TCP uses a three-way handshake to open connections
  • Flow control and windowing prevent receiver overload
  • Multiplexing lets multiple apps share one connection
TCP vs. UDP — quick reference
  • TCP: HTTP, HTTPS, FTP, SSH, SMTP
  • UDP: DNS, DHCP, VoIP, streaming, gaming
Exam tip Know the three-way handshake cold: SYN → SYN-ACK → ACK. This appears on nearly every Network+ and CCNA practice exam. Also be ready to identify whether a given protocol uses TCP or UDP.
